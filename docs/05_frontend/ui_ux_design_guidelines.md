# 06. UI/UXデザインガイドライン

本プロダクトのUI/UXを設計する上での基本原則とガイドラインを定義する。
プロダクトの目的である「ボトルネックの検知」と「ネクストアクションへの誘導」を達成し、開発者体験を向上させるための指針とする。

## 1. コアフィロソフィー

* **監視ツールではなく、診断ツールである:** 単一のスコアで評価・監視することが目的ではない。チームが自らの開発プロセスを理解し、改善の仮説を立てるための診断情報を提供することを最優先とする。

* **スコアではなく、傾向とバランスを重視する:** 数値そのものよりも、時間経過による「傾向」の変化や、各指標間の「バランス（トレードオフ）」に注目を促す設計を心がける。

* **問いから、行動へ:** すべてのデータ表示は、ユーザーが「なぜこうなっているのか？」という問いを立て、具体的な「次の行動」に移るための出発点でなければならない。

## 2. 設計原則

### 2.1. コンテキストを最優先し、安易な比較を避ける

* **NG:** チームやリポジトリをランキング形式で表示し、競争を煽るようなデザイン。

* **OK:** チームやリポジトリ単体の時間的な推移を表示する。フィルタリング機能により、ユーザー自身が意味のある比較軸（例：類似プロジェクト間）を選択できるようにする。

### 2.2. 指標のトレードオフを可視化する

* **NG:** Four Keysの各指標を個別のページやタブでバラバラに表示する。

* **OK:** 「デプロイ頻度 (Throughput)」と「変更障害率 (Failure Rate)」など、トレードオフの関係にある指標を常に同じ画面内に並べて表示し、両者のバランスが直感的に理解できるようにする。

### 2.3. すべての指標からドリルダウン可能にする

* **NG:** 数値やグラフを表示するだけで、その内訳がわからない。

* **OK:** ダッシュボード上のあらゆる指標（例：「CI実行時間」のグラフ）をクリックすると、その根拠となるデータ（例：「実行時間が長いジョブ一覧」）にドリルダウンできる。最終的にはCI/CDツールの元ログ(`url`)へシームレスに遷移できるようにする。

### 2.4. 「単一の大きなスコア」を避ける

* **NG:** 画面の中央に「ヘルススコア：85点」のような単一の数値を大きく表示し、それが絶対的な目標であるかのような印象を与える。

* **OK:** 複合的なスコアはあくまで参考情報として扱い、表示するとしても目立たないようにする。

## 3. 基本的なUIコンポーネント

* **メインダッシュボード:**

  * フィルタ（期間、プロジェクト、リポジトリ）を画面上部に配置。

  * Four Keysのトレンドを示すラインチャートを並べて表示する。

* **ドリルダウンビュー:**

  * 指標の内訳をテーブル形式で表示する。

  * テーブルの各行は、ソート・フィルタリング可能にする。

  * 各行から、CI/CDツールの該当ログへの外部リンクを提供する。
